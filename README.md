# Frame

An open-source Android slideshow / screensaver for the **Meta Portal Go** (Android 10 / API 29)
that shows your **Google Photos** shared albums. It runs as the Portal screensaver, has an
on-device setup screen (scan the album's QR code or paste the link), and adds photo-frame
touches like a clock, captions, cinematic motion, and ambient color.

> Project/repo name: **PortalFrame** (`com.portalhacks.frame`). The app displays as **Frame**.

**Just want to set it up on a Portal?** See the end-user **[Install & User Guide](INSTALL.md)**
(install a provided APK, add your album, turn it on as the screensaver). The rest of this README
is for developers.

## What it does / architecture
- **`FrameDreamService`** â€” a `DreamService` registered as Portal's screensaver
  (`screensaver_components`). It is a **thin trampoline**: on `onDreamingStarted` it launches
  `MainActivity` and calls `finish()`. It does NOT render the slideshow itself.
- **`MainActivity`** â€” the full-screen interactive slideshow (`KEEP_SCREEN_ON` +
  `SHOW_WHEN_LOCKED`/`TURN_SCREEN_ON`). This is what's actually on screen.
- **`SlideshowController`** â€” shared crossfade slideshow over images in `app/assets/slides/`.

### Why the trampoline (important)
A self-hosted UI *inside* the DreamService flickered badly on Portal: the device's ambient
manager kills an interactive in-dream window ~1s after it starts (`DREAM_FINISHED` in logs),
the idle device immediately re-dreams, and the cycle repeats â€” so it "went in and out", never
advanced past ~slide 3, and showed black teardown frames. Running the slideshow as a normal
foreground Activity (which keeps the screen on) avoids the dream lifecycle entirely: idle â†’
dream fires â†’ launches Activity â†’ Activity holds the screen â†’ tap finishes it â†’ idle â†’ repeat.

## Install & run on Portal Go
```
ADB=~/Library/Android/sdk/platform-tools/adb

# install
$ADB install -r app/build/outputs/apk/debug/app-debug.apk

# (A) run the kiosk slideshow Activity directly
$ADB shell am start -n com.portalhacks.frame/.MainActivity

# (B) make it the screensaver (back up the original first!)
$ADB shell settings get secure screensaver_components          # save this value
$ADB shell settings put secure screensaver_components com.portalhacks.frame/com.portalhacks.frame.FrameDreamService
$ADB shell settings put secure screensaver_enabled 1
$ADB shell am start -n com.android.systemui/com.android.systemui.Somnambulator   # trigger now to test

# restore Portal's stock Superframe screensaver
$ADB shell settings put secure screensaver_components com.facebook.alohaapps.launcher/com.facebook.aloha.app.home.touch.HomeDreamService
```

## Google Photos shared album (live)
The slideshow can pull from a public Google Photos **shared album** link. The official
Library API can no longer read your library (deprecated 2025-03-31), so `GooglePhotosSource`
scrapes the public share page: it follows the `goo.gl` â†’ `photos.google.com/share` redirect
and extracts the `lh3.googleusercontent.com` photo URLs that carry real dimensions, then
requests screen-sized images via a `=w1600` suffix. `ImageLoader` downloads them on a
background thread with a disk + memory cache (first loop downloads; later loops are instant
and survive reboots). The app shows bundled samples instantly, then hot-swaps to the album.

Set / change / clear the album over ADB (no rebuild):
```
ADB=~/Library/Android/sdk/platform-tools/adb
# set
$ADB shell am broadcast -n com.portalhacks.frame/.ConfigReceiver --es url "https://photos.app.goo.gl/XXXXXXXX"
# clear (revert to bundled samples)
$ADB shell am broadcast -n com.portalhacks.frame/.ConfigReceiver --es url ""
```
The album is re-fetched each time the screensaver launches, so adding photos to the shared
album in Google Photos makes them appear on the next idle cycle. If the album is unset,
empty, or unreachable, the bundled sample slides are shown instead.

## Set the album on-device by QR (no laptop)
For a non-technical user, the album can be set without ADB by scanning a QR code with the
Portal's camera â€” tap the **Frame** app icon on the Portal home (opens `SettingsActivity`).

> The Portal launcher shows sideloaded apps only if they ship a launcher **icon** and **label**.
> The home icon ("Frame") launches the Compose **`SettingsActivity`** â€” the setup/settings
> screen. It hands off to the Java **`PhotosActivity`** for the camera scanner / manual link
> entry. `MainActivity` (the slideshow) runs via the screensaver or a long-press, not as a
> separate launcher icon.

1. On your phone, open **Google Photos** â†’ open the album â†’ **Share** â†’ **Create link** and
   show its **QR code**. (The album must be shared by link, or the frame can't read it.)
2. On the Portal, open the **Frame** app. It shows the album currently playing (if any),
   including its title and a preview of the first photo.
3. Tap **Add album** / **Change album**, allow the camera once. Make the QR **fill your phone
   screen at full brightness** and hold it **~0.5 m** away so it's **sharp** and **fills the
   white box** â†’ "Album set âœ“". The slideshow picks it up on its next launch.
4. **Stop showing photos** clears the album and reverts to the bundled samples.

**No QR handy?** On the Photos screen tap **Enter link manually** (or **Can't scan? Type the
link** on the scanner) to type/paste the link with the on-screen keyboard. Same validation and
storage as the scan.

> Camera notes (Portal Go front camera): it's **fixed-focus** (a QR held closer than ~0.4 m is
> blurry â†’ hold it ~0.5 m away) and **ultra-wide** (so the QR must fill the box). The scanner
> picks the largest preview (1280Ã—720), decodes full-frame + mirrored + a centre crop with
> `TRY_HARDER`, and â€” because a bright phone screen otherwise blows out to white until
> auto-exposure slowly adapts â€” meters the **centre box** and biases exposure **âˆ’2 EV** so the
> QR is readable immediately. See `openCamera`/`tryDecode` in `PhotosActivity`.

(Slideshow gestures: short **tap** = dismiss, **swipe** = prev/next, **long-press** = open
Photos setup â€” a secondary way in. `PhotosActivity` is also launchable over ADB for dev:
`adb shell am start -n com.portalhacks.frame/.PhotosActivity`.)

The same screen has slideshow settings â€” **Seconds per photo**, **Shuffle photos**, and
**Transition speed** â€” persisted to the same prefs and read by `SlideshowController` on its
next start. QR (not OCR) is used because the share link ends in a random, case-sensitive
token; OCR would routinely mistype it. The decoder is ZXing's pure-Java `core`, vendored at
`app/libs/zxing-core-3.5.3.jar`. Scanning needs the
`CAMERA` permission (granted on first use). The ADB `ConfigReceiver` path above still works as
a developer fallback.

Videos are excluded. `GooglePhotosSource.parse` segments the page into media items
(boundary `["AF1Qip<key>",["https://lh3...`) and treats an item as a video if it either
contains a `video-downloads.googleusercontent.com` URL **or** carries neither a still
filesize element nor camera EXIF (transcoded clips). Each photo's capture date is read from
the item's timestamp + timezone offset and shown in the lower-right corner (location data is
not present in the share page, so only the date is shown).

Photos that don't match the screen aspect are NOT zoomed/cropped. `ImageLoader.composeFill`
renders each photo fit-centered (whole photo visible) over a blurred, center-cropped copy of
itself that fills the letterbox bars and matches the photo's edge colors (blur = downscale to
1/16 + box blur, upscaled, with a light scrim). Photos that already fill the screen hide the
blur.

Caveat: the scrape is unofficial and can break if Google changes the share-page format â€”
the regex lives in `GooglePhotosSource`. Verified working against a real album 2026-06-13:
8 photos (1 video + 1 duplicate excluded), portrait photos shown fit-centered with measured
340px blurred fill bars on each side, swipeable.
